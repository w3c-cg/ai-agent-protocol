# Meeting Assets for AI Agent Protocol CG Meeting

**Date**: 2026-06-03

## Quick Recap

The W3C AI Agent Protocol Community Group meeting on June 3, 2026 focused on an introduction to **Larry Lewis's Agent Trust Protocol (ATP)** and updates to the AI Agent Protocol CG draft documentation. Larry presented ATP as an open-source trust metadata layer for the agentic web, intended to address security and trust gaps in existing agent frameworks by providing verification and trust evidence across platforms. He explained that ATP is not meant to replace agent frameworks or agent communication protocols, but to sit between them and Web standards as a common layer for agent trust verification, using technologies such as decentralized identifiers (DIDs) and verifiable credentials (VCs).

The group discussed possible collaboration between ATP and the AI Agent Protocol CG because both efforts address agent identification, verification, trust, and communication. Sean then presented recent updates to the CG's protocol draft, especially the DID method work for agent identity. The updated draft includes a Web-based agent DID method, support for domain-name and path-type DIDs, path-bound public-key fingerprint verification, and basic DID Document service entries for agent description, handle, and message services. The meeting ended with discussion of implementation details, encrypted communication, and security considerations for the proposed protocols.

## Discussion Topics

### I. Opening and Agenda

Song welcomed new participants to the meeting and introduced the agenda. He explained that the meeting would include a presentation from Larry Lewis on agent trust protocols, a discussion of recent changes to the next version of the CG protocol document, and review of relevant issues and pull requests in the CG GitHub repository.

Song also briefly introduced his role as co-chair of the AI Agent Protocol Community Group and invited Larry to introduce his organization before presenting the main technical topic.

### II. Agent Trust Protocol Presentation

Larry Lewis, founder of **Sovereign Incorporated**, presented the **Agent Trust Protocol (ATP)** to the group. He described ATP as an effort to provide a missing trust layer for the agentic web.

Larry explained that many current agent frameworks provide useful orchestration, tool use, and communication capabilities, but often lack a standardized way to exchange trust metadata, verify agent identities, or attach verifiable evidence to agent behavior across platforms. ATP is intended to address that gap by adding a trust verification layer that can work across different vendor systems and protocol stacks.

Key points from the presentation included:

- ATP is designed as an **open-source trust metadata layer**.
- It is intended to complement, not replace, existing agent frameworks and protocols.
- It focuses on verification, trust evidence, and metadata that can help agents and platforms decide whether another agent or action should be trusted.
- It uses Web and decentralized identity concepts such as **DIDs** and **verifiable credentials**.
- Larry proposed **ATP 1.x** as a public specification that could be shared for W3C review and possible future standardization discussion.

### III. ATP and AI Agent Protocol CG Collaboration

The group discussed how ATP could relate to the AI Agent Protocol CG's work. Song noted that there is meaningful overlap between the two efforts, particularly around:

- Agent identification
- Agent verification
- Trust establishment
- Agent-to-agent communication
- Security metadata and evidence

At the same time, the group recognized that the efforts may operate at different layers. ATP appears to focus on trust metadata and evidence, while the AI Agent Protocol CG draft is currently focusing on agent identity, DID-based verification, discovery, and communication-related protocol building blocks.

George asked whether ATP already has a specification document. Larry confirmed that a specification exists and said he was willing to share it with the group after the meeting.

Song also asked about W3C membership and whether an ATP-related Community Group had been established. Larry said that he had joined several W3C groups and had requested the creation of an ATP group, but that final approval was still pending. The participants agreed to continue the coordination discussion by email.

### IV. Agent Protocol DID Implementation Updates

Sean presented recent updates to the AI Agent Protocol CG draft protocol document, focusing on the DID-based implementation for agent identity.

The updated protocol draft includes a Web-based DID method for agents, **did:wba**, which is extended from **did:web** and adapted for agent-oriented interoperability. Sean explained that the updated method improves security by binding a public-key fingerprint into the DID path for path-type DIDs, making the relationship between the DID and the controlled private key more explicit and verifiable.

The draft supports two main DID forms:

- **Naked domain-name DID**, used for a whole domain or domain-level service identity.
- **Path-type DID**, used for specific users, agents, or sub-identities under a domain, with a path-bound public-key fingerprint in the default profile.

Sean also described DID Document service entries used by the draft, including:

- **AgentDescription**, pointing to an agent description document.
- **ANPHandleService**, representing a handle service entry used by the external ANP ecosystem.
- **ANPMessageService**, representing a message service entry used by the external ANP ecosystem.

The protocol draft also follows existing RFC-based mechanisms for cross-platform authentication over HTTP, including HTTP Message Signatures and content digest binding.

The updated protocol draft is available at:

https://w3c-cg.github.io/ai-agent-protocol/protocol.html

Group members and interested participants are encouraged to review the draft and discuss issues or pull requests in GitHub:

https://github.com/w3c-cg/ai-agent-protocol/

### V. Implementation and Security Considerations

The final discussion focused on implementation details and security considerations for the proposed protocols. Topics included how DID-based authentication should be verified, how encrypted or end-to-end encrypted communication might be supported, and how ATP's trust metadata model could complement the CG's identity and communication work.

Larry indicated that he would review the shared AI Agent Protocol CG draft and provide feedback, especially on areas related to cryptographic and encrypted communication. The meeting ended shortly after this discussion because of time constraints and participant scheduling conflicts.
